## Implementation notes

Firebase Authentication is the identity provider. Recipe data remains under `app_data/enterprises/{enterprise_id}/users/{firebase_uid}`, while administrator membership data is maintained under `enterprise_directory`. The Firebase session endpoint performs an idempotent directory upsert so existing authenticated users are backfilled on their next session.

Saved-recipe integration tests verify that recipe listing and deletion are scoped by both enterprise ID and Firebase UID, and that accounts without enterprise membership cannot access enterprise recipe storage.

FCM registration, test delivery, and token deletion require a Firebase bearer token. The backend derives token ownership from the verified Firebase UID rather than accepting a client-supplied user ID.

Administrator dashboard smoke coverage verifies synchronized member retrieval, role-based denial for regular members, enterprise-membership enforcement, and the static dashboard's user rendering and empty-state contract.

Household profiles are stored under `app_data/enterprises/{enterprise_id}/users/{firebase_uid}/profiles/{profile_id}`. Each profile has independent preferences and cooking history, while the existing account-level saved recipe collection remains shared by the household. The profile page provides profile creation and active-profile selection.

## Release notes draft

GlucoPlate supports Firebase sessions with Google and email/password sign-in, cloud-backed recipes, authenticated notifications, enterprise administration, and household profiles with separate preferences and cooking history. Apple sign-in remains before the authentication milestone is complete.

- Saved recipes follow the user across devices.
- Notification tokens are tied to authenticated users.
- Profiles can support family or household preferences.
- Enterprise users are enrolled by access code and appear in the administrator directory.

## Implementation notes

Firebase Authentication is the identity provider. Recipe data remains under `app_data/enterprises/{enterprise_id}/users/{firebase_uid}`, while administrator membership data is maintained under `enterprise_directory`. The Firebase session endpoint performs an idempotent directory upsert so existing authenticated users are backfilled on their next session.

Saved-recipe integration tests verify that recipe listing and deletion are scoped by both enterprise ID and Firebase UID, and that accounts without enterprise membership cannot access enterprise recipe storage.

FCM registration, test delivery, and token deletion now require a Firebase bearer token. The backend derives token ownership from the verified Firebase UID rather than accepting a client-supplied user ID, and users cannot test or delete another account's device token.

Administrator dashboard smoke coverage verifies synchronized member retrieval, role-based denial for regular members, enterprise-membership enforcement, and the static dashboard's user rendering and empty-state contract.

## Release notes draft

GlucoPlate supports Firebase sessions with Google and email/password sign-in, backend ID-token verification, cloud-backed user recipe data, authenticated Firebase notification tokens, enterprise access-code enrollment, synchronized administrator directory membership, and dashboard smoke coverage. Apple sign-in and household profiles remain before the milestone is complete.
